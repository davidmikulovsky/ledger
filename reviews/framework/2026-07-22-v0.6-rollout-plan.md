## 0. The one idea everything hangs on

You said it yourself: the real culprit isn't any single missing feature — it's that **every step is manual** (ask, fetch, commit, log) and **nothing tells you what you don't know**. So the whole plan has a single spine, and every feature you asked for is a *reader* of that spine:

> **The scan engine.** One script that reads Ledger's units + live-scans every real project (git, README, changelog, decision log, dates), compares declared-state vs. reality, and writes one machine-readable file: `ledger-state.json`.

Build that once, and:

- the **dashboard** is just a viewer of that JSON,
- the **gap keywords** are just filters over that JSON,
- the **weekly/monthly checkup** is just "run the scan + notify if anything went red,"
- the **compliance catch-up report** (your PS1) is just "print the red/amber items for one project as a to-do,"
- **projects notifying Ledger** (your PS2) is just "the scan detects new commits/log entries and pulls them in."

Nothing is magic; it's all one spine and five thin readers. That's what makes it buildable, consistent, and easy to follow instead of a pile of one-off scripts.

```
                         ┌─────────────────────────┐
   _unit.md front-matter │                         │
   + live git/file scan  │   ledger scan  ───▶      │  ledger-state.json
   of each real project   │   (the spine)           │  (health + flags + drift)
                         └─────────────────────────┘
                                     │
        ┌──────────────┬─────────────┼──────────────┬───────────────┐
        ▼              ▼             ▼              ▼               ▼
   Dashboard      gap keywords   weekly/monthly   compliance     pull project
  (local web)    (audit/gaps)     checkup+notify   catch-up .md    updates → Ledger
```

## 1. What the scan engine actually computes

For each unit it produces a record. This is the data model the dashboard and every skill read — worth agreeing on the shape now.

```jsonc
{
  "generated": "2026-07-22T18:40:00+03:00",
  "units": [
    {
      "unit": "thann",
      "domain": "business",
      "parent": "business/egraine",        // link, not nesting (see §4)
      "type": "maintenance",
      "declared": { "status": "active", "priority": "medium", "urgency": "low" },
      "source": "/Users/mklvsky/AI/PROJECTS/Thann",
      "git": {
        "present": true, "branch": "main",
        "last_commit": "2026-07-22", "ahead": 3, "dirty": true,
        "locks": ["index.lock", "HEAD.lock"], "remote": "github.com/…/thann.git"
      },
      "docs": { "readme": "2026-07-22", "changelog": null, "decision_log": "2026-07-22" },
      "movement": "moving",                 // moving | stalling | blocked | dormant | ended
      "health": "red",                      // green | amber | red | grey
      "flags": [
        "DRIFT: project README says priority=high/urgency=high; Ledger says medium/low",
        "SOURCE-MISMATCH: unit source points at pen-drive xlsx, real repo is PROJECTS/Thann",
        "GIT-LOCK: live index.lock present — next commit will block"
      ],
      "verified": "2026-07-22"
    }
  ],
  "portfolio_flags": [
    "UNTRACKED: PROJECTS/Ozuvox on disk, no unit",
    "UNTRACKED: PROJECTS/project-cars on disk, no unit",
    "NO-GIT: Medex tracked active/high but has no git repo"
  ]
}
```

**How the colors and movement are decided (the rules the dashboard shows):**

| Signal | Rule |
|---|---|
| `health: red` | a hard violation or contradiction: git missing on an active unit, declared-vs-project drift, source path missing/wrong, live lock blocking commits, `disputed: true` unresolved. |
| `health: amber` | a soft gap: changelog/README stale vs. last commit, `verified:` date older than the cadence, uncommitted changes, missing canonical file. |
| `health: green` | conforms to the standard, recently verified, no drift. |
| `health: grey` | `dormant` or `ended` — intentionally idle, not a problem. |
| `movement: moving` | a commit or ledger entry within the "active window" (default 14 days). |
| `movement: stalling` | status is `active` but no commit/entry within the window — *the silent-death detector*. |
| `movement: blocked` | status is `blocked` (carries the blocker text from the unit). |
| `movement: dormant / ended` | from status. |

`stalling` is the single most valuable output — it catches "I forgot this project was moving and it quietly stopped," which is exactly your problem.

**Runs on your Mac**, where the repos live (git can't be inspected from the cloud). It only ever *reads* projects and only *writes* `ledger-state.json` + Ledger's own unit files — the boundary holds.

## 2. The expanded skill grammar (CLI keywords)

Everything extends the single `ledger` verb surface (ADR-0006's design principle — no competing skills). New verbs in **bold**:

```
ledger scan                          → refresh ledger-state.json (the spine; run before anything)
ledger audit                         → human-readable gap report (ships the designed-but-unbuilt verb)
ledger gaps [keyword]                → filtered red/amber list: urgent | stalling | blocked
                                          | noncompliant | untracked | nogit | drift | locks
ledger dashboard                     → (re)generate + open the local web dashboard
ledger check <project>               → "what changed but wasn't logged/committed" (consistency)
ledger doctor <project>              → downloadable catch-up .md: how to get compliant (your PS1)
ledger report <project> [depth]      → glance | brief | full | audit   (unchanged)
ledger report since:<date>           → time-window: what moved since a date (last week/month/year)
ledger portfolio [urgent|active|all] → cross-project rollup (unchanged)
ledger sync <project>                → check→update→log against real source (unchanged)
ledger log <project> "<update>"      → append an update (unchanged)
ledger new <project> <path>          → scaffold a standard-compliant unit + repo skeleton
ledger onboard <path>                → detect an untracked on-disk project and propose a unit
```

**Keyword-first gap-finding is the `gaps` verb.** `ledger gaps stalling` → every active-but-quiet project. `ledger gaps drift` → every unit whose declared state contradicts its project. `ledger gaps untracked` → on-disk projects with no unit. That's your "point at real gaps quickly with keywords."

## 4. Parent → child, built for spin-off (your Medex/Thann-can-be-sold requirement)

This is the most important model change, and your framing settles it: **ventures are separate repos precisely because any of them might be sold or spun out.** So Ledger must model them as *independent units linked to a parent*, never as content *nested inside* the parent. Concretely:

**Change:** replace the parent's `sub_units:` nesting with a flat unit + a link field.

```yaml
# business/thann/_unit.md   (promoted to its own top-level unit)
unit: thann
parent: business/egraine          # a LINK, not containment
relationship: distributed-brand   # egraine is Nordic distributor, not owner of the repo
source: /Users/mklvsky/AI/PROJECTS/Thann
```

```yaml
# business/egraine/_unit.md
portfolio_members:                # a rollup view, not ownership of their files
  - business/thann
  - business/medex
```

**Why this is exactly right for you:**

- **Spin-off is a one-line edit.** Sell Thann → delete `parent: business/egraine` (or repoint it to a new owner) and remove it from Egraine's `portfolio_members`. Thann's own repo, history, and ledger are untouched — it just detaches. No file surgery, no lost history.
- **The dashboard still shows the Egraine group together** (it groups by `parent`), so you keep the "parent + its ventures" view without hard-coupling them.
- **It matches the filesystem** (sibling repos) instead of fighting it, which kills the source-mismatch class of bug.

This becomes **ADR-0007 (group representation & spin-off model)** — see §9.

Paired with it, **ADR-0008 (child-safety / write boundary)**: a session working inside a child or sibling project treats parent/sibling folders as **read-only**, and any destructive op (bulk delete, `git rm`, folder moves) requires your explicit go-ahead + a verified backup — never a side effect. This is the mirror of ADR-0001 and the direct fix for the Egraine 48-file near-miss.

## 5. The git-lock & folder-access policy (your explicit ask)

A new guideline `LEDGER/guidelines/git-and-access.md` + **ADR-0009**, covering three things:

**(a) Access is yours alone.** Default stance for any AI session is **read-only**. Before the *first* `git init`, or any write/commit/delete in a project, the session must **ask you for permission**, naming the exact folder. **Only you grant or revoke** folder access (via the desktop "Add folder" / access grant). No session assumes write rights. This is written as a hard rule, extending ADR-0001.

**(b) Preventing lock cruft in the first place.**
- One session per repo at a time — never run two Cowork sessions doing git on the same folder (concurrent `index.lock` is the main cause).
- The device-bridge sandbox blocks delete/rename by default; that's what orphans locks and breaks `git init`. So write/delete permission is granted deliberately per project (per (a)), not left ambiguous.
- Commit in small real steps, finish git operations before ending a session (a half-finished commit is what leaves a live lock).

**(c) Cleaning locks safely (the runbook).**
- A lock is **stale** if no git process is running against that repo. Then `rm .git/index.lock` / `rm .git/HEAD.lock` is safe.
- If the sandbox can't delete (permission not granted), the fallback is the `_to_delete/` pattern: move the lock there renamed `*.stale`, and **you empty `_to_delete/` yourself** periodically (it's the one thing the AI can't finish).
- `ledger scan` **surfaces every live lock as a red flag** so they never accumulate invisibly again — Thann and project-cars have live locks *right now*, and the scan would have shown that on day one.

## 7. Periodic automation & checkups (weekly / monthly, adjustable)

Honest constraint first: a scheduled task runs in a **fresh cloud session** and can only touch your Mac's repos **while your desktop app is connected** with folders granted. Scheduled runs can't force your laptop open. So the design degrades gracefully:

- **Primary — scheduled Cowork task** (weekly or monthly, adjustable): on fire, if the device bridge is available it runs `ledger scan`, writes the JSON, and sends you a **digest + push/email notification** — "3 reds, 1 newly stalling (Ozuvox), Thann still drifting." If the bridge isn't available that run, it notifies you to run `ledger scan` and reads the last committed snapshot so the digest still has something.
- **Cadence is one setting** you change any time ("make it monthly" → I adjust the schedule).
- **Optional — local launchd** on your Mac for a pure offline scan that writes the JSON on a timer even with no Claude session, so the dashboard is always current when you open it. (A small plist I give you; you install once.)

Recommendation: start with the **weekly Cowork digest** (that's the "tap me on the shoulder" you're missing), add the local launchd later if you want the dashboard self-refreshing.

## 8. Answers to your PS questions

**PS1 — the compliance catch-up report.** Yes, this is `ledger doctor <project>`. It reads the scan's flags for one project and emits a downloadable `.md`: what's missing, why it matters, and the exact ordered steps to get compliant (init git / add the canonical changelog / fix the status line / point the unit at the right source). It's the friendly, per-project version of the audit — a "get well" checklist. Cheap, because it's just the red/amber items formatted as a to-do.

**PS2 — projects notifying Ledger.** Good idea, and the clean way to do it is **pull, not push**: don't let a project write into Ledger (that keeps the boundary simple and avoids two-way coupling). Instead, `ledger scan` *detects* new dated entries in a project's own `ledger.md`/decision log and new commits, and folds a summary into that unit's Ledger ledger automatically. If you want it snappier than the next scan, a project skill can drop a one-line `.ledger-signal` file in its own folder ("2026-07-22 shipped Phase 1"); the next scan reads it, records it, clears it — a lightweight outbox. Primary = pull-on-scan; signal-file = optional. Either way Ledger stays the single aggregator and nothing violates the boundary.

**PS3 — how to make skillsets shareable (the real mechanisms).** Three levels, cheapest to most robust:

1. **`.skill` file** — a skill is just a folder (`SKILL.md` + resources) zipped. I can hand you one via the file panel; your org settings decide if it can be saved to your account. Good for a single skill, one person.
2. **A shared skills folder** — skills placed in a synced location are picked up across your sessions. Good for just you, across devices.
3. **A plugin (the right answer for a team).** A plugin bundles skills + commands + MCP connectors into one installable unit distributed from a marketplace. **The whole Ledger admin surface should become one "Ledger" plugin** — the `ledger` skill(s), the scan engine, the dashboard generator, the guideline docs — so you (and later your AI-LAB team) install it once and everyone has the same verbs and the same standard. This is also exactly what your AI-LAB team-rollout needs. You already have the `cowork-plugin` and `skill-creator` skills available to build and package it.

So: build the skills, then **package them as the Ledger plugin** in the final phase. That makes them shareable *and* keeps them location-agnostic so they run inside a project or inside Ledger.

**PS4 — noted, all of §7 from the audit is in the plan (Phase 2 below).**

**PS5 — walkthrough of "Next (enforcement)" from the audit:**

- *Ship audit + consistency + time-window.* These are the three readers in §2 (`audit`, `check`, `report since:`). Once the scan engine exists they're small: each is a formatter over `ledger-state.json`. "Enforcement" means these run routinely (Phase 4 automation) so gaps surface without you asking.
- *ADR on group representation.* That's §4 / ADR-0007 — the parent-link, spin-off-ready model. It's "enforcement" because once the model matches disk, the scan can *automatically* verify parent links and catch a venture that wandered off.
- *Child-safety rule.* That's §4 / ADR-0008 — children treat parents read-only, destructive ops need your ok. It's enforcement because it prevents the whole class of accidental-deletion incidents structurally, not by remembering to be careful.

**PS6 — should ALFA+OMEGA be its own git repo?** **Yes — do it.** It's your canonical governance source, it's currently the *least* protected copy (no version history at all), and you already need a v2. Make it a git repo, structured for versioning:

```
ALFA+OMEGA/            (its own git repo)
  v1/  project-standard.md  tracking-protocol.md  workflow.md  checklists.md
  v2/  … (the update you're planning — starts as a copy of v1, edited)
  CHANGELOG.md         what changed between versions and why
  LATEST → v1          a pointer/symlink to the current canonical version
```

Then Ledger and AI-LAB **reference** `ALFA+OMEGA/LATEST/` instead of holding independent copies — which resolves the three-copy drift risk in one move (see PS7). This is **ADR-0010 (canonical guidelines repo + versioning)**.

**PS7 — walkthrough of "Later (hygiene)":**

- *Single source of truth for guidelines.* Today three byte-identical copies exist with no link between them (ALFA+OMEGA, LEDGER/guidelines, AI-LAB/docs/guidelines). The moment one is edited they diverge silently. Fix (via PS6): ALFA+OMEGA becomes the versioned canonical repo; the other two stop being copies and become references/symlinks to `ALFA+OMEGA/LATEST/`, or thin pointer files that say "canonical lives here, vX." One edit, one place, no drift.
- *Canonical filename convention.* Every project currently names the same thing differently (`ledger.md` vs `Decision Log.md` vs `.xlsx` vs `docs/decisions/` vs `technical/decision-log.md`). Pick one canonical set — proposal: `CHANGELOG.md` (the "what"), `decisions/` or a single `decision-log.md` (the "why"), `README.md` with the three-axis status line, optional `CLAUDE.md`. Bake it into `ledger new` so new projects are born compliant, and `ledger doctor` nudges old ones toward it. Hygiene, not urgent — but it's what makes the scan's file checks reliable everywhere.
